# `pmt fmt` Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Add a `pmt fmt` formatter for `.pmc`: a wholesale reprint from a lossless CST that `parse()` now builds (and lower-copies to the existing AST), with a label/command alignment model, comma-group + comment layout, and a batch CLI (`--check`, `-`).

**Architecture:** `parse()` builds one lossless CST; the compiler lowers a *copy* to the existing `Program` AST (flavor C1) so `flatten`/`ir::lower`/optimizer/codegen are UNCHANGED (guarded by a parity test). `fmt` reads the CST and pretty-prints, library-side, as `format(source) -> Result<String, CompileError>`; `cli/fmt.rs` is the only printer/FS-toucher.

**Tech Stack:** Rust; `serde`/`serde_json` + `proptest` (dev) only — no new deps. Read the design spec `docs/superpowers/specs/2026-07-07-pmc-fmt-design.md` and the audit `.superpowers/sdd/fmt-audit.md` — every task's rules come from the spec; the audit grounds the C1 refactor.

## Global Constraints

Every task's requirements implicitly include these. Copy the binding ones into each task's reviewer prompt.

- **No new dependencies.** Gates at every commit: `cargo test --workspace` fully green, `cargo clippy --workspace --all-targets -- -D warnings`, `cargo fmt --check` (fmt first). Commit only on a CLEAN tree (`git status` empty) with green gates.
- **Parity is a BINARY gate (Tasks 3+):** after any commit that touches the parser/CST/lower path, `parse(x) == parse_legacy(x)` and identical accept/reject across the FULL `.pmc` corpus (stdlib + goldens + fixtures) AND every parser grammar test, AND `cargo test --workspace` green. If parity is not green, STOP — do not build later tasks on an unproven parser.
- **Zero compiler behavior change:** `pmt compile` and `pmt lint` output byte-identical to pre-change; `-O0` bit-identity holds (golden suite); `PMC_LANG_VERSION` stays `0.2`. The lexer's `WithoutComments` significant-token stream is byte-identical to today.
- **Thin renderer:** library returns typed results; only `cli/fmt.rs` prints and touches the filesystem. `format(source) -> Result<String, CompileError>` (NOT `-> String`).
- **fmt makes ZERO token changes** — pure whitespace/blank-line/comment-position transform. No `left()` stripping (grammar 0.2 already rejects it), no leading-zero rewrite (lint's job), no call re-mangling.
- **Conventional commits**, scope per crate: `feat(post-machine):`, `feat(core):`, `test(post-machine):`, `docs(fmt):`, `polish(post-machine):`. **No AI/Claude attribution footers.**
- **Published docs are forge-agnostic** (no issue/PR numbers, no host URLs): `docs/fmt.md`, `docs/cli.md`, `docs/lint.md`, `docs/language.md`, `README.md`, code comments. Internal artifacts (this plan, the spec, the ledger) are unrestricted.
- Do **NOT** merge or push; the branch is left for the user's review.

### Task 1: Lexer `WithComments` mode

**Files:** Modify `crates/post-machine/src/lexer.rs`.

**Interfaces:**
- Produces: a way to lex retaining comments as trivia. Suggested: `lex(source)` stays (`WithoutComments`, byte-identical); add `lex_with(source, mode: LexMode) -> Result<Vec<Token>, CompileError>` where `LexMode::WithComments` interleaves `TokenKind::Comment(Comment)` tokens at their source positions. `Comment { text: String, kind: CommentKind (Line|Block), own_line: bool }` — `text` verbatim incl. delimiters; `own_line` = only whitespace preceded it on its physical line (the cursor knows the column at comment start).

**Steps (TDD):**
- [ ] Write failing tests: `WithComments` on `"// a\nleft; // b\n/* c */ right;"` yields the significant tokens WITH interleaved `Comment` tokens carrying text/kind/own_line/span (line comment own_line=true for `// a`, own_line=false for the trailing `// b`; block comment kind=Block). And a test that `lex(src)` (`WithoutComments`) is **byte-identical** to today for a multi-comment program (the existing `comments_are_skipped` test must still pass unchanged).
- [ ] Implement: in the `/` arm (lexer.rs ~97–124), instead of `continue`-discarding in `WithComments` mode, push a `Comment` token with span + kind + own_line; `WithoutComments` keeps discarding. Preserve the unterminated-block-comment and lone-`/` errors in both modes.
- [ ] Run tests green; full workspace suite green (nothing else consumes the new mode yet).
- [ ] Commit.

**Constraints:** `WithoutComments` significant-token stream unchanged (token-equivalence). No new deps.

### Task 2: CST node types

**Files:** Create `crates/post-machine/src/cst.rs`; wire `mod cst;` in `lib.rs`.

**Interfaces (Produces):** the lossless node set (spec "Architecture → Comments as trivia" + audit §6). Derive `Debug, Clone, PartialEq, Eq`. Sketch:
- `Cst { items: Vec<TopItem> }` (file level).
- `TopItem::Import(ImportCst) | TopItem::Namespace(NamespaceCst { name, name_span, items: Vec<TopItem> }) | TopItem::Function(FunctionCst)`.
- `FunctionCst { exported: bool, name, name_span, line, col, body: Vec<BodyItem> }`.
- `BodyItem::Statement(StatementCst) | BodyItem::Nested(FunctionCst)`.
- `StatementCst { labels: Vec<Label>, items: Vec<Item>, line, span, label_break: bool }` — reuse the existing `Label`/`Item`/`Successor`/`Builtin` types from `parser.rs` (re-export or move to a shared module) so `lower_cst` can hand them straight to `Program`. `label_break` = the author put a newline after the final `:` (own-line label — spec).
- Trivia (comments) are carried on nodes at their positions — decide a representation (e.g. `leading: Vec<Comment>` + `trailing: Option<Comment>` per statement/item, or an interleaved `Vec<CstElement>`). Choose the shape that makes the pretty-printer's comment rules (Task 7) and `lower_cst`'s trivia-drop (Task 3) simplest; document the choice in the module header.

**Steps (TDD):**
- [ ] Write a compile-level test that constructs a small `Cst` by hand and asserts field access + `PartialEq` round-trips (`clone == self`).
- [ ] Implement the types with doc comments citing `docs/language.md` for the grammar shapes.
- [ ] Workspace green (types unused elsewhere yet — a `#[allow(dead_code)]` on the module is acceptable INTERIM, removed when Task 3 consumes it).

**Constraints:** no behavior change; types only.

### Task 3: `parse_cst` + `lower_cst` + rewire `parse` + `parse_legacy` + parity  **[HIGH RISK]**

**Files:** Modify `crates/post-machine/src/parser.rs`; add a parity test (in `parser.rs` `#[cfg(test)]` or a `tests/parser_parity.rs`).

**Interfaces:**
- `parse_cst(tokens: &[Token]) -> Result<Cst, CompileError>` — the lossless parser. Same control flow, scope tracking, duplicate checks (`DuplicateName`/`DuplicateLabel`), and grammar errors as today; constructs CST nodes preserving namespace-block boundaries + reopening, top-level order, in-body statement/nested interleaving, `label_break`, and comment trivia (from `WithComments` lex). Carries spans/lines VERBATIM.
- `lower_cst(cst: &Cst) -> Program` — walks the CST doing exactly what `parse()` does today (stamp `ns` tags from the enclosing namespace path; collect nested fns into `Function.nested` out of body order; flatten namespaces to the flat `Program.functions`/`.imports`), dropping trivia and `label_break`, copying spans/lines through verbatim. No re-checking (validated in `parse_cst`).
- `parse(tokens) -> Result<Program, CompileError>` = `parse_cst(tokens).map(|c| lower_cst(&c))`. **Public signature unchanged** — its ~9 callers (compile + optimizer/codegen/ir test helpers) are untouched.
- `parse_legacy(tokens) -> Result<Program, CompileError>` = the CURRENT parser implementation, frozen and RENAMED, kept as the parity oracle. Mark it `#[doc(hidden)]`; note in a comment it exists solely for the parity guard and its keep-vs-delete is a review decision.

**Steps (TDD) — smallest safe increments:**
- [ ] Freeze the current parser as `parse_legacy` (rename, no logic change). Workspace green. Commit.
- [ ] Write the parity test FIRST (it will fail to compile until `parse` is rewired): for every `.pmc` in the corpus (embedded `std.pmc`, `crates/post-machine/tests/**` fixtures, the golden `sum.pmc`/`ty.pmc`) AND every source string in the parser's own grammar test suite, assert `parse(lex(src)) == parse_legacy(lex(src))` and that both are `Ok`/`Err` together with the same `CompileErrorKind`. (Gather corpus paths with a small helper; keep the source strings list next to the grammar tests.)
- [ ] Implement `parse_cst` + `lower_cst`; rewire `parse`. The moment it compiles, run the parity test across the WHOLE corpus + `cargo test --workspace`. A span/line mismatch means `lower_cst` recomputed instead of copying — fix by carrying the field through. Iterate until parity is GREEN and the whole workspace is green.
- [ ] Remove the Task-2 interim `#[allow(dead_code)]` (CST is now consumed).
- [ ] Commit only when parity + workspace are fully green.

**Constraints (BINDING):** parity is a binary gate (Global Constraints). `parse()`'s public signature and all callers unchanged. Duplicate checks stay in `parse_cst`. Do NOT delete `parse_legacy` (review decision). Compile/lint byte-identical; goldens green; `PMC_LANG_VERSION` 0.2.

**This is the C1 milestone — a clean, parity-proven commit here is the safe checkpoint before any fmt code.**

### Task 5: Pretty-printer — statements + label/command alignment

**Files:** Modify `crates/post-machine/src/fmt/` (add a statements/alignment submodule).

**Interfaces (Consumes):** the CST statements; produces aligned statement lines.

**Steps (TDD):** implement the spec's *Label / command alignment* rules exactly. Test cases (assert exact output):
- [ ] Command column from the widest inline label: a body with `1: right;` + `check(1,2);` → command column driven by `1:` (P=2 → col 4); commands aligned; unlabeled at the column.
- [ ] Right-aligned padding: widest `11111` + `12` → col 8, `12` padded left (spec worked example — reproduce it byte-for-byte).
- [ ] Tab-stop round-up: a 2-digit widest label → command column 8 (`P+2=5` rounds up to 8).
- [ ] Own-line label (`label_break`) that FITS → right-aligned to the colon column, command on the next line at the column; one that OVERFLOWS → hangs at 1-space, command on next line at the column (spec worked example — reproduce byte-for-byte).
- [ ] fmt never auto-breaks a label (a long inline label stays inline, column grows).
- [ ] Statement splitting: `left; right;` → two lines.
- [ ] Base-indent interaction: inside `namespace ns { f() {...} }`, base body indent 8 dominates when labels are narrow.
- [ ] Re-run the Task-4b harness over these inputs; workspace green. Commit.

**Constraints:** exact spec conformance; idempotent (harness). This is the meatiest fmt task — prefer a small pure helper computing the command column from the CST body, unit-tested independently.

### Task 7: Pretty-printer — comments

**Files:** Modify `crates/post-machine/src/fmt/`.

**Steps (TDD):** implement the spec's comment rules. Tests:
- [ ] Leading group above a function preserved at the node indent (the `std.pmc` doc-comment shape).
