# `pmt` Shell Completion — Design

**Date:** 2026-07-06
**Repo:** `mellonis-workspace/machines/toolchains`
**Status:** implemented (zsh); bash/fish are a documented follow-on

## 1. Problem

`pmt` has 8 top-level subcommands now (`compile`, `asm`, `link`, `dis`,
`tape`, `run`, `ir`, and `completions` itself, the last added by this
change), two of which nest a sub-subcommand (`tape build`/`tape show`,
`ir graph`), each with its own flags and file-extension-specific
positional arguments. None of that is discoverable at the shell prompt
without shell completion. `pmt`'s arg parsing (`cli::Args` in
`cli/mod.rs`) is hand-rolled by deliberate minimal-dependency choice — no
clap — which means a completion script cannot be generated by a
framework the way `clap_complete` gives a clap-based CLI completions for
free. It has to be authored by hand, which creates a real risk: the
script drifts from the flags the parser actually accepts as either side
changes independently.

## 2. The crux: one registry, one drift guard

The whole design rests on a single idea: **describe the CLI surface once,
as data, in `crates/post-machine/src/completions/registry.rs`**, and
render every shell's script as a pure text transform over that data
(`crates/post-machine/src/completions/zsh.rs` today). Nothing about a
shell's syntax leaks into the registry, and nothing about a specific
subcommand's flags is retyped per shell. A **drift guard**
(`crates/post-machine/tests/completions_registry.rs`) then cross-checks
the registry against the real parser and the optimizer's own data, so a
future change to `cli/build.rs`/`cli/inspect.rs`/`cli/run.rs` or to the
optimizer's pass list can't silently leave the registry — and therefore
every rendered script — stale.

### 2.1 Registry shape

```rust
pub struct CommandSpec {
    pub path: Vec<String>,       // ["compile"], ["tape", "build"], or [] for root
    pub flags: Vec<FlagSpec>,
    pub positional: Positional,  // None | One(hint) | OneOrMore(hint)
}

pub struct FlagSpec {
    pub name: String,                    // "-o", "--emit-ir", or "--fno-" (a prefix)
    pub help: String,                    // short completion-menu blurb
    pub kind: FlagKind,
    pub repeatable: bool,                // -L, -l
    pub exclusive_group: Option<String>, // -O0/-O1; --tape/--tape-block
    pub requires: Option<String>,        // gated flag (no current entry needs this)
}

pub enum FlagKind {
    Boolean,                        // -g, --trace
    Value(ValueHint),                // --name value OR --name=value
    OptionalEqualsValue(ValueHint), // bare --name, or --name=value only — never --name value
    SuffixFamily(Vec<String>),      // --fno-inline, --fno-dce, ... (one flag per choice)
}

pub enum ValueHint { Text, Choices(Vec<String>), File(FileHint), Directory }
pub struct FileHint { pub extensions: Vec<String>, pub dirs: bool }

pub enum Positional { None, One(PositionalHint), OneOrMore(PositionalHint) }
pub enum PositionalHint { File(FileHint), Choices(Vec<Choice>), Text }
```

`registry()` builds a flat list of `CommandSpec`s — one root (`path:
[]`), one per top-level leaf (`compile`, `asm`, `link`, `dis`, `run`,
`completions`), and one per nested leaf (`tape build`, `tape show`, `ir
graph`). There is no separate "group" node for `tape`/`ir` in the data —
the zsh renderer derives which top-level names are groups purely by
noticing which `path[0]` values have depth-2 children, and the root's own
subcommand-name choice list is derived from the other entries' `path`,
not hand-duplicated. This is deliberate: every place a name or a choice
list could be retyped is instead computed from one place, because a
retyped copy is exactly the kind of thing that rots.

### 2.2 Three flag shapes the hand-rolled parser actually has, not two

Reading `cli/build.rs`/`cli/inspect.rs`/`cli/run.rs` closely surfaced
three genuinely different value-delivery shapes, not the one "flag takes
a value" shape a naive registry would assume:

- **`Value`** — the ordinary case. `cli::Args::value` accepts both
  `--name value` and `--name=value` uniformly. Almost every value flag
  (`-o`, `-L`, `--max-steps`, `--tape-block`, ...) is this.
- **`OptionalEqualsValue`** — `--emit-ir[=STAGE]` is the *only* current
  example, and it is genuinely different: `take_emit_ir` in
  `cli/build.rs` checks for the bare flag OR a `--emit-ir=` prefix, but
  never calls `Args::value`. A space-separated `--emit-ir lowered` is
  **not** accepted — `lowered` is left behind as a stray positional and
  the compile fails with "compile takes exactly one input". The zsh
  renderer emits this as `--emit-ir=-[...]:...` (zsh's `=-` suffix means
  "value optional, and if given, only accepted joined with `=`"), which
  matches the real parser precisely.
- **`SuffixFamily`** — `--fno-<pass>` is the other genuine oddity: it
  isn't `--fno- value`, it's a *family* of complete flags, one per
  optimizer pass (`--fno-inline`, `--fno-dce`, ...), scanned by a raw
  `strip_prefix("--fno-")` loop that treats the suffix itself as the
  pass name. The registry expands this into one literal flag per pass at
  render time (`registry::expand`), and the choices come from
  `optimizer::pass_names()` — the same list `--emit-ir=after:<pass>`'s
  choices are built from — not a second retyped copy. This one detail is
  the centerpiece of the drift guard (§3): the CLI issue that asked for
  this feature paraphrased the pass names with underscores
  (`check_fold`, ...); the real names, straight from
  `optimizer::mod::PIPELINE`, are hyphenated (`check-fold`, ...). Reading
  the source instead of the issue text is what caught this, and the
  guard exists so the next person doesn't have to re-derive it by hand.

### 2.3 Read vs. write: why output paths aren't extension-filtered

`compile -o`, `asm -o`, `link -o`, `tape build -o`, and `run
--save-tape-block` all write a *new* file whose name the user is
choosing, not opening an existing one — filtering the shell's file list
down to files matching an extension that (usually) doesn't exist yet on
disk would just hide the very path the user is about to type. So every
output-path flag uses `ValueHint::File(FileHint { extensions: vec![],
.. })`, which the zsh renderer turns into a plain, unfiltered `_files`.
Every *input* positional and *input* flag (`compile`'s `.pmc`, `dis
--map`'s `.pmx.map`, `run --tape-block`'s `.pmt`, ...) IS
extension-filtered via `_files -g "*.ext"`, since those files need to
already exist and match a real extension. This read/write split is a
deliberate simplification, not an oversight: it means `compile -O src.pmc
-o <TAB>` doesn't try to be clever about whether `-S` changed the
implied output extension from `.pmo` to `.pma` — it just shows you the
directory.

**Open follow-up, not done in this pass:** a nicer `-o` experience would
derive the *default* output name from the sibling input already on the
command line (e.g. `pmt compile sum8.pmc -o <TAB>` suggesting `sum8.pmo`
the way `cli/build.rs::out_path` computes it internally via
`input.with_extension(...)`). That needs the zsh action to inspect
`$words` for the already-typed positional, which is a materially fancier
completion function than anything else in this script. Flagged during
review as something to fold into this branch later if wanted, not
implemented here.

## 3. The drift guard

`crates/post-machine/tests/completions_registry.rs` is the mechanical
check the issue asked for. What it actually verifies, and how:

1. **`optimizer_pass_names_match_the_registry_exactly`** — the
   registry's `--fno-<pass>` `SuffixFamily` choices and `--emit-ir`'s
   `after:<pass>` choices are asserted equal, in order, to
   `optimizer::pass_names()`. This is the single most valuable check in
   the file: it is the exact thing that had already drifted once (in the
   issue's own prose), and it's a zero-cost equality assertion because
   both sides ultimately read the same function.
2. **`every_registry_flag_is_accepted_by_the_real_parser`** — for every
   `CommandSpec` and every literal flag spelling `registry::expand`
   produces, actually invoke `cli::execute` with that spelling on that
   subcommand's path and assert the resulting error (if any) does not
   contain `"unknown flag"`. `cli::Args::positionals` is the one place in
   the hand-rolled parser that rejects an unrecognized dashed token, with
   that exact wording — so this is a real parser probe, not a second
   hand-maintained list being compared to itself.
3. **`root_flags_are_accepted_by_the_real_top_level_dispatch`** — same
   idea for `--help`/`-h`/`--version`, which `cli::execute_with` handles
   before any subcommand dispatch.
4. **`top_level_subcommands_match_the_maintained_list_cli_help_and_the_real_parser`**
   — the registry's top-level name set (derived from `CommandSpec.path`)
   is compared against a maintained literal list in the test, against
   the CLI's own top-level `--help` text, and each name is probed the
   same way as flags (must not produce `"unknown subcommand"`) — plus a
   sanity check that a made-up name IS rejected that way, proving the
   probe itself is meaningful.
5. **`emit_ir_stage_choices_are_all_accepted_by_the_real_stage_check`** —
   `take_emit_ir` validates the stage string itself (`lowered` / `final`
   / an `after:` prefix) independently of the flag scan; this probes that
   predicate directly with every choice the registry advertises.

**The one honest gap**, stated plainly in the test file's own header
comment: nothing here can detect a real flag the registry is *missing*.
`cli::Args` offers no reflection over `cli/build.rs`'s match arms — there
is no framework parse tree to walk, no attribute macro output to diff
against. Catching an added-to-the-parser-but-not-the-registry flag
depends on the PR that adds it also touching the registry, same as any
other hand-rolled CLI without a generator. The guard's job is to make the
things that *can* be checked mechanically — pass names, flag spellings,
subcommand names — actually checked, not to pretend to a completeness the
architecture doesn't support.

`crates/post-machine/tests/completions_zsh.rs` covers a different axis:
does the *rendered zsh text* actually work as zsh. It shells out to a
real `zsh` binary (skipping with a printed note if none is on `PATH`) to
run `zsh -n` (pure syntax check) and to load the script under a real
`compinit` in a hermetic `zsh -f` session, asserting `whence -w _pmt`
resolves. `_arguments`/`_describe` refuse to run at all outside the
actual completion-widget machinery ("can only be called from completion
function") — calling them directly with a hand-mocked `$words`/`$CURRENT`
was tried during development and fails for that reason, so genuine
interactive candidate-correctness (does pressing Tab after `pmt compile
foo.pmc -O` actually offer `-O0`/`-O1`?) was checked by hand with a pty
feeding real keystrokes, not automated. The content-level unit tests
inside `src/completions/zsh.rs` (every subcommand, every flag, the right
`-g` glob per positional, the mutual-exclusion parens, the `--fno-`
expansion) are what stand in for that in the automated suite.

## 4. The zsh script model

Standard "git-style" nested completion (`man zshcompsys`): the root
function's `_arguments -C '1: :->cmds' '*::arg:->args'` consumes the
subcommand name into `$state`, `_describe` offers it from a `subcommands`
array built straight from the registry's root positional choices, and a
`case $words[1] in ... esac` dispatches to a per-subcommand function.
Because the root call used `-C`, each subcommand's own nested
`_arguments` call automatically re-bases its positional numbering to
start *after* the words already consumed — this is what lets every leaf
function complete its own `1`/`*` positional without knowing how deep in
the tree it sits. `tape` and `ir` repeat the same two-state pattern one
level deeper (their own `_arguments -C` is required for the same reason,
applied to their child, `tape build`/`tape show`/`ir graph`).

Per-flag rendering follows directly from `FlagKind` (§2.2): `Boolean` →
`'name[desc]'`; `Value` → `'name[desc]:msg:action'`; `OptionalEqualsValue`
→ `'name=-[desc]:msg:action'` (the `=-` suffix is exactly zsh's spelling
of "optional value, `=`-joined only"); `SuffixFamily` → one
`'prefixchoice[desc]'` line per choice. `exclusive_group` renders as the
`(other1 other2)name[...]` prefix zsh uses for mutual exclusion —
`-O0`/`-O1` and `--tape`/`--tape-block` both use this. Extensions render
as `_files -g "*.ext"` (single) or `_files -g "*.(ext1|ext2)"` (`dis`'s
dual `.pmo`/`.pmx`); directories (`link -L`) render as `_files -/`; a
literal choice list (`--emit-ir`'s stages, `completions`'s shell names)
renders as a parenthesized word list with any literal colon
backslash-escaped (`after:inline` → `after\:inline` — zsh's `_arguments`
spec grammar treats an unescaped colon as a field separator even inside
parentheses).

## 5. Why zsh only, this pass

Zsh has the richest completion model of the three (`_arguments`,
`_describe`, extension-aware `_files -g`) and was the primary target the
issue asked for. `bash` and `fish` are structurally mechanical once the
registry exists (bash: `complete -F` plus `compgen -f -X` for coarser
extension filtering; fish: declarative `complete -c pmt` statements per
flag) — but "mechanical" still means writing and testing two more
renderers, and this pass was already large (registry + guard + zsh +
docs). Shipping zsh solidly, proven by both the drift guard and a real
`zsh -n`/`compinit` check, was judged better than shipping three shallow
renderers. `completions::Shell` already has all three variants and
`parse_shell` accepts all three names, so `pmt completions bash`/`fish`
name themselves explicitly in a "not implemented yet" error rather than
being rejected as unknown shells — the follow-on is adding
`completions::bash`/`completions::fish` render functions plus their own
content-level unit tests, no registry or CLI-wiring change required.

## 6. The forthcoming `pmt lint` and `pmt build` — pre-specified additions

Two subcommands are inbound that are **not** registered as active
registry entries in this change — the drift guard must stay green
against the real, current 8 subcommands, not a frozen-forever 7. This
section is what to add when they land, so nothing gets forgotten.

### 6.1 `pmt lint`

From the issue this is scoped against: `pmt lint PATH... [--exclude
PATH]... [--allow CODE]... [--fix [--force]]` — exit 0 clean, 1
findings/errors. Its harder cases and the registry field each maps to
are exactly why `Positional::OneOrMore`, `FileHint.dirs`, `repeatable`,
and `requires` all exist already, even though nothing in the *current*
registry sets `dirs: true` or `requires: Some(_)`:

```rust
CommandSpec {
    path: vec!["lint".into()],
    positional: Positional::OneOrMore(PositionalHint::File(FileHint {
        extensions: vec!["pmc".into()],
        dirs: true,       // lint also accepts directories, not just files
    })),
    flags: vec![
        FlagSpec::value("--exclude", "skip a path (repeatable)",
            ValueHint::File(FileHint { extensions: vec!["pmc".into()], dirs: true }))
            .repeatable(),
        FlagSpec::value("--allow", "allow a lint code (repeatable)", ValueHint::Text)
            .repeatable(),
        FlagSpec::boolean("--fix", "apply fixes"),
        FlagSpec::boolean("--force", "overwrite without confirmation").requires("--fix"),
    ],
}
```

This is proven to typecheck and behave as expected today by
`completions::registry::tests::registry_shape_can_express_the_forthcoming_pmt_lint`
in `crates/post-machine/src/completions/registry.rs` — that test builds
exactly this shape (not wired into `registry()`) and asserts the
`dirs`/`repeatable`/`requires` fields come out right. When `lint` lands
for real: add this `CommandSpec` to `registry()`'s list, add
`lint_spec()` alongside the other `fn ..._spec()` constructors, add a
short one-line gloss to `top_level_help`, and extend
`completions_registry.rs`'s `EXPECTED_TOP_LEVEL` list. The zsh renderer
needs **no** change for `Positional::OneOrMore`/`repeatable`/`dirs:
false` (already exercised by `link`'s `.pmo...` and `-L`/`-l`) — the one
piece of new rendering work is `dirs: true`'s directory-plus-extension
completion (today's `file_action` ignores `dirs` since nothing sets it
live; it'll need something like zsh's `_alternative` combining `_files
-g "*.pmc"` with `_files -/`) and, optionally, rendering `requires` as a
completion-time hint (`_arguments` has no first-class "B requires A", so
this would most likely show up as descriptive text on `--force` rather
than conditional suppression).

### 6.2 `pmt build`

Tracked in issue #11 as a one-step compile+link driver — an 8th (now 9th,
post-`completions`) subcommand with its own flag union, most likely a
superset combining `compile`'s optimization/debug flags with `link`'s
library flags, plus its own output-path flag. No concrete flag list
exists yet to sketch a `CommandSpec` for (unlike `lint`, which the issue
that spawned it already specifies precisely); when it lands, the
addition is mechanically the same shape as any other leaf: a
`build_spec()` constructor, an entry in `registry()`'s list, a
`top_level_help` gloss, and an `EXPECTED_TOP_LEVEL` update.

## 7. Non-negotiables this design respects

- **No new dependencies.** No clap, no regex/glob/walkdir — the registry,
  the expansion logic, and the zsh renderer are all hand-rolled string
  building, matching the project's existing minimal-dependency stance.
- **Thin-renderer rule.** `completions::registry` and `completions::zsh`
  never print; `cli::completions` (the CLI-facing wrapper) only parses
  the shell-name argument and hands the rendered string back in
  `CliOutput` for `bin/pmt.rs` to print, exactly like every other
  subcommand.
- **Published-docs policy.** `docs/cli.md`'s new `pmt completions`
  section and this design doc's own body avoid issue/PR numbers and
  forge URLs in the parts that mirror published-doc content; this
  section (§6) and the rest of this internal spec are unrestricted, per
  the workspace's published-docs-reference policy.
